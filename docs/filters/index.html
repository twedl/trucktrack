Trucktrack filter pipeline

Cleaning and segmenting truck GPS traces

Jesse Tweedle

Pipeline overview

The problem

Raw truck GPS streams arrive dirty:

  • Devices re-emit buffered readings verbatim
  • Bad fixes imply physically impossible motion
  • Signal drops create ragged temporal gaps
  • Long idles need to be separated from driving
  • Some “stops” are actually traffic, not real stops

The pipeline turns a noisy per-truck stream into clean, segmented trajectories.

Stage order

Each stage is per-truck and independent across trucks.

What each stage adds

Stage Drops rows Adds column
filter_stale_pings yes —
filter_impossible_speeds yes —
split_by_observation_gap (via min_length) segment_id
split_by_stops (via min_length) segment_id, is_stop
filter_traffic_stops no (reclassifies is_stop)

Why this order

  • Stale before speed — re-emissions have tiny dt against the real prior row; they’d look like huge speeds, and speed would drop them for the wrong reason.
  • Speed before gap — removing impossible points first keeps gap segments from being artificially fragmented.
  • Stop after gap — stops are detected inside a temporally continuous window; gaps break that assumption.
  • Traffic last — needs the is_stop labels that split_by_stops produces.

filter_stale_pings

What / why

Some devices buffer readings and later re-emit an earlier record with only the timestamp advanced.

A sequence like T1 → T2 → T3 appears, where T3 has the same (lat, lon, speed, heading) as T1 but a later time.

import trucktrack

clean = trucktrack.filter_stale_pings(df, window=5)

Per truck, each row is compared bit-exactly against the last window distinct rows; matches are dropped.

Bit-identical re-emission

Before — source (blue) and stale copy (red)

After — stale row dropped

What changed

The 08:04:00 source and the 08:05:30 re-emission share (lat, lon, speed, heading) exactly. The intermediate 08:05:00 ping has different lat/lon, speed, and heading — that’s what a real subsequent observation looks like.

The filter’s only job is removing the duplicate; the rest passes through untouched.

What the map matcher sees

Before — matcher loops the block · After — straight northbound, no detour

Why the detour

The route runs through downtown Toronto on a divided one-way arterial. The raw trace forces the matcher to revisit the source position — but each carriageway is a separate one-way OSM way, so the matcher detours east, south on the opposite-direction street, then west to rejoin.

With the stale row filtered out, the matcher routes straight northbound — no detour, no back-tracking.

Params and gotchas

  • window tradeoff — too small misses re-emissions of older records; too large risks dropping legitimate revisits to a prior location.
  • speed == 0 exemption — a stopped truck legitimately emits many identical pings. Without this, a stationary row in the buffer would also spuriously flag a later moving ping that happens to pass through the same coords.
  • Partial match is kept — same lat/lon but different speed or heading is not a re-emission.
  • Nulls pass through — a null in any of the four compared fields never matches.
  • Trucks are independent — truck B’s fresh ping cannot match truck A’s buffer.
  • Time is not part of the match — only (lat, lon, speed, heading) is compared. The re-emission’s timestamp can be any value later than the source.

filter_impossible_speeds

What / why

GPS glitches produce fixes that imply motion faster than any truck can move.

clean = trucktrack.filter_impossible_speeds(df, max_speed_kmh=200.0)

Per truck, sorted by (id, time). For each row, compute Haversine distance and dt against the last kept point — not the previous row. If the implied speed exceeds the threshold, drop the row and hold the anchor.

Threshold is on computed speed (distance ÷ time), not the device-reported speed column: glitched GPS often emits a self-consistently wrong reported speed.

Off-route spike on 401 westbound

Before — spike 5.5 km north · After — dropped

Why the detour

Truck cruising westbound on Highway 401 at 60-second cadence. Ping 25 lands ~5.5 km north of the corridor — that’s ~330 km/h of implied motion from its predecessor, well past the 200 km/h ceiling.

The matcher tries to honor the off-route fix and bends the route north to reach it. With the spike dropped, ping 26 is evaluated against ping 24 (the last kept anchor) and the route stays on 401.

Params and gotchas

  • max_speed_kmh — default 200. Converted internally to m/s.
  • Anchor held across drops — consecutive spikes collapse against the last clean fix; one bad point cannot drag subsequent evaluation with it.
  • First valid point per truck always kept — no prior anchor to compare against.
  • Nulls pass through without advancing the anchor — a null lat/lon/time can’t be evaluated, so the row survives and the next row still compares against the last clean fix.
  • dt == 0 is safe — duplicate timestamps do not divide by zero; the row is kept.

split_by_observation_gap

What / why

A trajectory is only physically meaningful within a window of continuous observation. Yard dwells with the device shut off, privacy toggles, coverage holes — any of these leave a large dt between two otherwise-unremarkable pings.

from datetime import timedelta

segmented = trucktrack.split_by_observation_gap(
    df,
    gap=timedelta(minutes=2),
    min_length=0,
)

Per truck, start a new segment_id whenever the time delta to the previous row exceeds gap.

Five rows straddling a 10-hour shutoff

Before — moving truck, dt jumps to 36060 s

After — segment_id splits at the gap

What changed

Truck cruising westbound at ~63 mph, heading around 256°. The device reports normally at 60-second cadence, goes dark for 10 hours, then resumes — further down the route, still moving. Every column looks like normal driving; only dt tells you anything happened. With a 2-minute threshold, the splitter inserts a new segment at the 36060-second row.

This is what makes the gap distinct from a stop: the pings aren’t clustered, speeds aren’t zero, headings aren’t frozen. A stop detector would see nothing; the gap splitter reads dt and segments.

What the matcher sees

Before — matcher invents a U-turn · After — split, two clean legs

Why the U-turn

A → near-B → A out-and-back trip. The device fell silent on the WB carriageway at the Hespeler Rd interchange and resumed 10 hours later on the EB carriageway, ~370 m away.

Without splitting, the matcher has no clue this isn’t a continuous trip and invents a U-turn through the on/off ramps. Splitting first ends one trip at ping (2) and starts a new one at ping (3); both legs match cleanly along their own carriageway.

Params and gotchas

  • gap: timedelta — the only tuning knob for segmentation.
  • min_length — drops whole segments shorter than N rows. Set aggressively and the output can be empty; in the test suite, min_length=20 on 15-row segments drops everything.
  • Tz-aware datetimes — handled identically to naive; no panic, same segment count.
  • Per-truck independence — vehicle A’s gaps cannot affect vehicle B’s segmentation.
  • Tiny thresholds fragment everything — a 1-second threshold on 30-second pings gives every row its own segment.
  • Space is irrelevant — the filter never looks at lat / lon. A truck that drove 500 km during a dropout and a truck that sat still are treated identically; only the time delta matters.

split_by_stops

What / why

Trucks idle: loading docks, rest breaks, driver swaps. A “stop” is a window of pings that stays inside a small diameter for long enough.

segmented = trucktrack.split_by_stops(
    df,
    max_diameter=50.0,            # meters
    min_duration=timedelta(minutes=2),
    min_length=0,
)

Within a truck, find maximal windows where all pings fit inside max_diameter and span at least min_duration. Label them is_stop = true; label everything else as movement.

Segments alternate movement → stop → movement.

Fuel rest at a 401 service area

Before — pings stack on the Petro Canada lot · After — dwell flagged

What the detector did

Truck pulls off Highway 401 westbound at the Comber Petro Canada, parks for 15 minutes, then re-enters 401 westbound. The detector finds the cluster: 16 pings inside a 50 m diameter spanning ~15 minutes.

One segment_id per phase (approach / dwell / departure), is_stop true on the middle segment. All rows preserved — rows are never dropped by the stop detector itself.

Params and gotchas

  • max_diameter (m) — maximum spatial extent for a window to count as stopped.
  • min_duration — minimum temporal span. Both must hold simultaneously.
  • min_length is asymmetric — drops short movement segments but keeps stop segments regardless. A 10-row movement segment vanishes at min_length=15; a 6-row stop survives.
  • Large diameter or duration → no stops — max_diameter=100_000 m with min_duration=24 h finds nothing in a normal day’s trace.
  • All rows preserved by default — useful for downstream filters (traffic) that need the original positions around each stop.

filter_traffic_stops

What / why

Not every detected stop is a real stop. Trucks also halt at red lights, in congestion, and behind queued traffic — all on the road corridor they were already following.

cleaned = trucktrack.filter_traffic_stops(
    df,                           # output of split_by_stops
    max_angle_change=30.0,        # degrees
    min_distance=10.0,            # meters
)

For each stop segment, compute the approach bearing (points before the stop) and the departure bearing (points after). If the angular change is below max_angle_change, the truck stayed on the same corridor — reclassify the stop as movement and merge segments.

Construction halt on 401 westbound

Before — blue leg / orange ring / orange leg · After — single continuous blue, no break

Contrast — real stop preserved

The Comber Petro Canada dwell from the previous slide. Same shape (cluster of pings, >2 min, <50 m), but the truck approaches from 401 and leaves at a sharp angle onto the driveway. Bearing change ≈ 90°, above threshold. Stop survives.

Params and gotchas

  • Requires split_by_stops output schema — expects segment_id and is_stop. Hard dependency.
  • max_angle_change (degrees) — default 30. Lower = stricter (more stops kept); higher = more reclassification.
  • min_distance (m) — minimum separation between point pairs used for bearing computation. Filters GPS jitter out of an otherwise noisy bearing signal.
  • Bearings from positions, not device heading — device heading can freeze or lag; positions are ground truth.
  • Segment IDs are renumbered — after reclassification, IDs are sequential again, so downstream code doesn’t see gaps in the segment_id space.
